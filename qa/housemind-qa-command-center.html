Severity definitions
P0 — Release blocker
Release cannot proceed. Requires immediate fix and re-test on physical A13 before any shipment. QA holds a hard block. Cannot be overridden by PM or Engineering.
P1 — Mitigation required
Release may proceed only after a documented mitigation plan is reviewed and approved by QA. Workaround must be communicated to affected users before deployment.
P2 — Tracked, non-blocking
Logged in Linear with target milestone. Does not block release. Reviewed in next sprint planning.
P3 — Nice to fix
Cosmetic, edge-case, or very low frequency. Backlogged. No release impact.

Triage flow

QA discovers issue on A13 or via automated test failure. Creates Linear ticket immediately with full repro steps, media, and build ID.
2
QA assigns severity using policy above. If any automatic P0 trigger is met → P0, no discussion required or permitted.

Escalation path

Silent deferral is forbidden. If a performance threshold is not met on A13, the team must hold a scope reduction discussion before the next release candidate is cut. Issues may not be carried forward silently.

Regression = P0 automatically. Any metric that degrades > 20% vs the previous passing build is a P0, even if it stays under the absolute threshold. Both absolute and regression conditions must be clean for a pass.

Escalation chain: QA block → Engineering Lead → CTO. PM cannot override QA at any level of the chain.

Linear configuration required: Labels P0, P1, P2, P3 must exist. Milestone "Release Gate" must exist. QA sign-off = merging label qa-approved to the release branch.
